# Semantic Memory — applycling

## Core Systems

- **Unified Application Profile:** `data/profile.json` is the single source of truth for profile, contact, and applicant fields. `profile_completeness()` returns one of 5 states (missing_contact → missing_resume → ready → enriched → complete). `DEFERRED_PROFILE_FIELDS` defines the deferred set. `data/applicant_profile.json` functions are deprecated wrappers. `applycling profile status` and `profile edit` CLI commands available.
- **Skills architecture:** 16 LLM prompts live in `applycling/skills/<name>/SKILL.md` with YAML frontmatter (name, description, inputs, output_file, model_hint, temperature). Loader in `applycling/skills/loader.py` — `load_skill(name).render(**kwargs)` validates inputs, renders via `str.format`.
- **Pipeline:** `applycling/pipeline.py` — `PipelineContext`, `PipelineStep`, `PipelineRun`. Linear deterministic flow: `role_intel → resume_tailor → profile_summary → format_resume → positioning_brief → cover_letter → email_inmail → fit_summary`. All callers (CLI, local workbench, OpenClaw) use this library API.
- **Additional capabilities:** `interview_prep` (composes `interview_prep` + `questions` skills) and `follow_up_outreach` (composes follow-up skills) — both shipped and usable today via CLI.
- **Queue/intake (partial):** `applycling/queue.py` — drives OpenClaw integration; not yet fully wired for multi-source intake.
- **LLM routing:** `applycling/llm.py` — ollama, anthropic, google, openai providers. API keys in `.env` (gitignored), loaded via `python-dotenv`.
- **Tracker abstraction:** `applycling/tracker/` — `get_store()` auto-detects backend: Notion (`data/notion.json`), SQLite (default, zero-config), or Postgres (`APPLYCLING_DB_BACKEND=postgres` with `DATABASE_URL`). Never call stores directly from CLI. PostgresStore uses psycopg v3 with `dict_row` factory, UUID primary keys, `migrate_old_status()` on all writes. Opt-in via `pip install .[postgres]`. Schema managed by Alembic (hand-authored raw-SQL migrations, no ORM); Docker Compose provides local Postgres 16. Single-user local mode seeded via `applycling/db_seed.py`.
- **Canonical state machine:** `applycling/statuses.py` — 11 states, 25 transitions, frozen dataclasses (`Status`, `StatusAction`). Single source of truth for all paths: CLI, UI, Telegram, API, MCP. `OLD_TO_NEW` maps legacy vocab (`tailored`, `interview`, `offer`, `inbox`, `running`, `generated`, `skipped`) to canonical states.
- **Local web workbench:** `applycling/ui/` — FastAPI + Jinja2 at `http://127.0.0.1:8080`. Job board, detail view, submit form, artifact serving, status transitions. Async fire-and-forget pipeline dispatch via `schedule_pipeline_run()` with retained `_background_tasks` references and belt-and-suspenders error surfacing. Meta-refresh on detail page when status is `generating`. Regenerate is a dedicated endpoint (`POST /jobs/{id}/regenerate`), not a status transition — gated to `new`/`reviewing`/`failed`.
- **Pipeline duplicate-prevention:** `persist_job=False` in `PipelineContext` — when the caller owns the job row (workbench), `run_add()` creates a transient `Job` with the caller's id but skips `save_job()`. No duplicate rows, no fragile delete/merge.
- **Artifact fallback:** `jobs_service.list_artifacts()` scans the package folder when `artifacts.json` is missing (Telegram/CLI-created jobs). `_INFER_KIND` maps filenames to artifact kinds.
- **Renderer:** `applycling/render.py` — markdown → HTML → PDF via Playwright/Chromium. `h3 em { float: right }` right-aligns dates.
- **Agent agnosticism:** `.agent/generate.py <agent>` generates entry-point files + hooks for 7 agents: Claude Code, Codex, Hermes, Cursor, Gemini CLI, Windsurf, OpenClaw. Memory files (`memory/`, `dev/`, `DECISIONS.md`) are portable across all agents. Architecture vision doc: `vision.md` (was `ARCHITECTURE_VISION.md`).
- **Telegram intake / onboarding:** Inbound Telegram messages reach applycling via the hosted Hermes profile on the VPS host (`~/.hermes/profiles/applycling/`). Hermes is outside Docker and calls the Dockerized applycling app through host-local `http://127.0.0.1:8080/api/forward`. `/api/forward` owns onboarding state (`new` → `confirming` → `active`) and user resolution by `telegram_id`. SOUL.md must not contain credential names or inspect env/files/logs/DB/package folders. Outbound delivery uses `applycling/telegram_notify.py`. Hosted provision/update: `scripts/setup_hosted_hermes.sh`; local/laptop setup script is legacy/dev-only.
- **Hosted dogfood deployment:** `docker-compose.prod.yml` targets a Kamatera VPS with Caddy TLS, Postgres, and bind-mounted `/opt/applycling/{data,output}` for config/profile data and durable artifacts. applycling/Postgres/Caddy run in Docker Compose under `/opt/applycling/app`; Hermes runs on the VPS host, not in Docker. Hosted UI auth uses HMAC-signed session cookies (`applycling_session`) with login at `/login`; `/healthz`, `/api/intake`, and localhost-only `/api/forward` are exempt.
- **Active-run guard:** Postgres enforces one running pipeline per user with partial unique index `idx_pipeline_runs_active` on `(user_id) WHERE status = 'running'`. `create_run()` uses atomic `INSERT ... ON CONFLICT ... DO NOTHING`; `check_active_run()` is UX-only. SQLite paths are no-ops.
- **Startup stale-run sweep:** Hosted/Postgres startup calls `sweep_all_running()` so orphaned active runs do not permanently block generation after restart.
- **Async workbench runs:** Submit/regenerate use `schedule_pipeline_run(job_id, user_id=...)` fire-and-forget. Hosted session routes pass the authenticated `user_id` through job creation, status updates, artifact lookup, active-run pre-checks, and `jobs_service.run_pipeline()` so Postgres workbench jobs stay scoped to their owner. Intake (multi-tenant) uses `PipelineContext.from_user_id()` + `run_add()` via `_run_scoped_pipeline()` with per-user error surfacing via `notify_error_to_user()`.
- **Health endpoint:** `/healthz` returns 200 when DB is reachable and 503 when unhealthy.
- **Hermes forwarding intake (multi-tenant):** `/api/intake` remains a secret-bound compatibility endpoint. `/api/forward` is the localhost-only Hermes relay endpoint for Telegram messages. It consumes `link CODE` before lookup, resolves existing non-deleted users by `telegram_id`, never auto-creates rows, preserves linked-user confirming corrections in `profile.pending_corrections` (capped at last 10), activates approved linked users via `APPROVAL_KEYWORDS` frozenset, builds `PipelineContext.from_user_id()` only after guards pass, then schedules `_run_scoped_pipeline()`. Unknown onboarding states fail closed with a restart prompt rather than falling through to active-pipeline behavior. Endpoint responses are bounded to relay fields only. State machine helpers live in `applycling/forward_endpoint.py`; state column `users.onboarding_state` ∈ {`new`, `confirming`, `active`} (migration `005_add_onboarding_state`, CHECK constraint, default `new`).
- **Forward host allowlist:** `verify_localhost()` accepts loopback + IPv4-mapped loopback plus comma-separated networks from `APPLYCLING_FORWARD_ALLOWED_SOURCES`. Required because the app runs in Docker behind userland-proxy: traffic from host-installed Hermes hitting `127.0.0.1:8080` reaches the container as the Docker bridge gateway, not loopback. `docker-compose.prod.yml` pins subnet `172.30.0.0/24` / gateway `172.30.0.1` and sets the allowlist default accordingly. Documented in `docs/deploy/DEPLOY.md`.
- **Web onboarding (session-gated):** `/onboarding` → `POST /onboarding/submit-resume` → `/onboarding/confirm` (`applycling/ui/routes.py`). The web onboarding surface is behind session auth; only API relay endpoints are auth-exempt. Server-side HMAC-SHA256 tokens (`_sign_onboarding_user_id` / `_verify_onboarding_token`, constant-time compare) replace raw `user_id` in URLs and forms — closes IDOR. Empty, malformed, or tampered tokens 403 on both GET and POST. Token secret resolves `APPLYCLING_ONBOARDING_TOKEN_SECRET` → `APPLYCLING_SESSION_SECRET` should become the GA direction; the old UI password fallback is legacy cleanup.
- **Web session auth:** `applycling/auth.py` provides PBKDF2-SHA256 password hashes and HMAC-signed session tokens. `SessionMiddleware` in `applycling/ui/__init__.py` is fail-closed in hosted Postgres mode: requires `APPLYCLING_SESSION_SECRET`, refuses `APPLYCLING_NO_AUTH`, and redirects unauthenticated browser requests to `/login`. `APPLYCLING_ADMIN_USER_ID` gates `/admin`; `/admin/invite` creates active users with generated passwords.
- **MCP server:** `applycling/mcp_server.py` exposes pipeline as MCP tools via FastMCP. `mcp serve` (stdio), `mcp config` (client JSON). Shipped tools: `add_job` (MCP-T1), `list_jobs(limit, status)` and `get_package(job_id)` (MCP-T2), `update_job_status(job_id, status, reason)` via `jobs_service.set_job_status()`, `interview_prep(job_id, stage, model, provider)` writes interview_prep.md, and `refine_package(job_id, feedback, artifacts, cascade, model, provider)` versions before overwriting (MCP-T3). Shared helpers in `applycling/package_actions.py` — `ConfigurationError`, no CLI/UI deps. `list_jobs` reads via `get_store().load_jobs()` with Notion→SQLite fallback. `get_package` returns bounded markdown artifact content with 50K file / 200K total byte caps. Architecture contract at `architecture/MCP.md`.
- **MCP alpha installer:** `scripts/setup_mcp.sh` supports the public `curl | bash` path and attaches the interactive `applycling setup` wizard to `/dev/tty`, because prompt input otherwise sees piped stdin and aborts with "Input is not a terminal".
- **Hosted workbench user scoping:** `jobs_service.run_pipeline(job_id, user_id=...)` must build `PipelineContext.from_user_id()` and pass it into `pipeline.run_add_notify()`. Without that explicit context, `run_add_notify()` uses local `data/profile.json` / `data/resume.md`, which caused a 2026-05 alpha incident where a hosted UI generation used the wrong person's profile data.
- **Hosted Telegram scoping:** `_run_scoped_pipeline()` in `applycling/ui/routes.py` must call `run_add_notify(url=ctx.job_url, context=ctx, persist_job=ctx.persist_job)`, not `run_add(ctx)`. Telegram/intake callers do not pre-create a workbench job row, so they should preserve `PipelineContext.from_user_id(...).persist_job=True`; only workbench-owned `jobs_service.run_pipeline()` uses `persist_job=False`.
- **Web-first Telegram linking:** Web/admin invite is the canonical account creation path; Telegram is a linked channel only. `/api/forward` handles `link CODE` first via `consume_telegram_link_code()`, then resolves existing non-deleted users by `telegram_id` without creating rows. Unlinked Telegram messages return link-code instructions and never store resume text, change onboarding state, charge quota, or dispatch pipeline work. Linked users can still use active URL handling and confirming approval/correction from Telegram.
- **Alpha duplicate-account remediation/linking:** `applycling users link-code --user-id ...` creates a one-time Telegram link code for an existing web/canonical user; Telegram `/api/forward` consumes `link CODE` to attach Telegram to that existing row. `applycling users merge --source-user-id ... --target-user-id ...` remains the admin repair path for already-split accounts. Merge refuses running pipelines, moves source jobs/runs/artifacts to the target, fills missing target profile/resume/channel fields from the source, then soft-deletes the source after clearing unique identity fields.
- **Canonical web profile/resume:** `/profile` and `/onboarding` edit the same current user row used by Telegram/workbench. Web users can upload PDF/DOCX/Markdown/text resumes; uploads are converted to stored text via `applycling.resume_import.extract_resume_text()`. Updating the resume always overwrites `users.resume` on the canonical user row, so linked Telegram and web sessions share the same source resume/profile.
- **Seamless onboarding follow-up:** Login redirects incomplete users to `/profile`; `/profile` shows a shared progress checklist and can generate one-time Telegram link codes without admin CLI.
- **Pre-flight resume guard:** All four remaining pipeline dispatch paths (`/api/forward` active, `/api/intake`, web `/submit`, web `/jobs/{id}/regenerate`) refuse to schedule a pipeline when the scoped user has no stored resume text. Guards run before daily-cap charge with surface-specific messages.

- **Artifact durability:** Hosted output artifacts are persisted through the `/opt/applycling/output` bind mount; package folders remain available across container rebuilds/restarts.
- **Two-layer LLM:** Hermes profile uses DeepSeek for routing; applycling pipeline uses Anthropic Claude for generation. Configs: `~/.hermes/profiles/applycling/config.yaml` (routing) vs `data/config.json` (generation).

## Key Patterns

- **Thin harness, fat skills:** Python is boring plumbing. Intelligence lives in `.md` files.
- **Library-first design:** `pipeline.run_add()` is the public contract. CLI is a thin wrapper.
- **Deterministic pipeline:** User intent is always "URL → package". No agent routing needed yet (context-based resolvers are planned).
- **Escaped braces in skills:** `{{` and `}}` render as literal `{`/`}` after `str.format`.
- **Conditional logic stays in Python:** Skills have no `if/else`. Caller pre-computes strings, passes as inputs.
- **MCP parity:** When a new pipeline capability ships, the corresponding MCP tool ships in the same PR. Pipeline and MCP surface must not drift.

## Known Gotchas

- **Skill name mismatch:** Loader raises `SkillError` if frontmatter `name` doesn't match directory name.
- **Profile summary header:** Must be `## PROFILE` (all caps) to match format template.
- **`storage.save_config()` merges:** Never call with partial keys unless merging is intent.
- **`_clean_llm_output()` required:** Strips code fences, preamble, leaked prompt markers from all LLM output.
- **`_profile_header_markdown()` builds static header:** Constructs the name/contact block from `profile.json` for resume output. Never let the LLM generate this section — always use the pre-computed static header.
- **Anthropic API key in repo .env:** That key is for the applycling pipeline only — never use it for Hermes subagent delegation, ad-hoc API calls, or any non-pipeline purpose. Hermes has its own keys in `~/.hermes/`.

## Hermes Profile

- **Profile:** `applycling` at `~/.hermes/profiles/applycling/` on the VPS host.
- **Gateway service:** hosted Hermes gateway is systemd-managed on the VPS (`hermes-gateway-applycling`). Older local launchd plist references are laptop/dev only.
- **Env vars:** no applycling secrets or DB/API keys in profile `.env`; `TELEGRAM_ALLOWED_USERS` is the allowlist.
- **SOUL.md:** Dumb relay that POSTs `{telegram_id, chat_id, first_name, message_text}` to `http://127.0.0.1:8080/api/forward`; terminal-only profile, no direct DB/file/browser/vision tools.
- **Operational order:** deploy Docker app + run Alembic migration first, then update SOUL/profile env, then restart hosted Hermes only when no pipeline is active.
