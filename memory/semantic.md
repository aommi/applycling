Async fire-and-forget pipeline dispatch via `schedule_pipeline_run()` with retained `_background_tasks` references and belt-and-suspenders error surfacing. Meta-refresh on detail page when status is `generating`. Regenerate is a dedicated endpoint (`POST /jobs/{id}/regenerate`), not a status transition â€” gated to `new`/`reviewing`/`failed`.
- **Pipeline duplicate-prevention:** `persist_job=False` in `PipelineContext` â€” when the caller owns the job row (workbench), `run_add()` creates a transient `Job` with the caller's id but skips `save_job()`. No duplicate rows, no fragile delete/merge.
- **Artifact fallback:** `jobs_service.list_artifacts()` scans the package folder when `artifacts.json` is missing (Telegram/CLI-created jobs). `_INFER_KIND` maps filenames to artifact kinds.
- **Renderer:** `applycling/render.py` â€” markdown â†’ HTML â†’ PDF via Playwright/Chromium. `h3 em { float: right }` right-aligns dates.
- **Agent agnosticism:** `.agent/generate.py <agent>` generates entry-point files + hooks for 7 agents: Claude Code, Codex, Hermes, Cursor, Gemini CLI, Windsurf, OpenClaw. - **Telegram intake / onboarding:** Inbound Telegram messages reach applycling via the hosted Hermes profile on the VPS host (`~/.hermes/profiles/applycling/`). Hermes is outside Docker and calls the Dockerized applycling app through host-local `http://127.0.0.1:8080/api/forward`. `/api/forward` owns onboarding state (`new` â†’ `confirming` â†’ `active`) and user resolution by `telegram_id`. - **Hosted dogfood deployment:** `docker-compose.prod.yml` targets a Kamatera VPS with Caddy TLS, Postgres, and bind-mounted `/opt/applycling/{data,output}` for config/profile data and durable artifacts. applycling/Postgres/Caddy run in Docker Compose under `/opt/applycling/app`; Hermes runs on the VPS host, not in Docker. - **Active-run guard:** Postgres enforces one running pipeline per user with partial unique index `idx_pipeline_runs_active` on `(user_id) WHERE status = 'running'`. `create_run()` uses atomic `INSERT ... ON CONFLICT ... DO NOTHING`; `check_active_run()` is UX-only. SQLite paths are no-ops.
- **Startup stale-run sweep:** Hosted/Postgres startup calls `sweep_all_running()` so orphaned active runs do not permanently block generation after restart.
- **Async workbench runs:** Submit/regenerate use `schedule_pipeline_run(job_id, user_id=...)` fire-and-forget. Hosted session routes pass the authenticated `user_id` through job creation, status updates, artifact lookup, active-run pre-checks, and `jobs_service.run_pipeline()` so Postgres workbench jobs stay scoped to their owner. Intake (multi-tenant) uses `PipelineContext.from_user_id()` + `run_add()` via `_run_scoped_pipeline()` with per-user error surfacing via `notify_error_to_user()`.
- **Health endpoint:** `/healthz` returns 200 when DB is reachable and 503 when unhealthy.
- **Hermes forwarding intake (multi-tenant):** `/api/intake` remains a secret-bound compatibility endpoint. `/api/forward` is the new localhost-only Hermes relay endpoint for all Telegram messages. It auto-creates users with `get_or_create_user_by_telegram()` (atomic upsert; soft-deleted rows surface a bounded 409 relay message, not a 500), stores resume text for new users, preserves raw confirming corrections in `profile.pending_corrections` (capped at last 10, echo truncated to 240 chars), activates approved users via `APPROVAL_KEYWORDS` frozenset, builds `PipelineContext.from_user_id()` before charging quota, then schedules `_run_scoped_pipeline()`. Unknown onboarding states fail closed with a restart prompt rather than falling through to active-pipeline behavior. Endpoint responses are bounded to relay fields only. State machine helpers live in `applycling/forward_endpoint.py`; state column `users.onboarding_state` âˆˆ {`new`, `confirming`, `active`} (migration `005_add_onboarding_state`, CHECK constraint, default `new`).
- **Forward host allowlist:** `verify_localhost()` accepts loopback + IPv4-mapped loopback plus comma-separated networks from `APPLYCLING_FORWARD_ALLOWED_SOURCES`. Required because the app runs in Docker behind userland-proxy: traffic from host-installed Hermes hitting `127.0.0.1:8080` reaches the container as the Docker bridge gateway, not loopback. `docker-compose.prod.yml` pins subnet `172.30.0.0/24` / gateway `172.30.0.1` and sets the allowlist default accordingly. - **Web onboarding (session-gated):** `/onboarding` â†’ `POST /onboarding/submit-resume` â†’ `/onboarding/confirm` (`applycling/ui/routes.py`). The web onboarding surface is behind session auth; only API relay endpoints are auth-exempt. Server-side HMAC-SHA256 tokens (`_sign_onboarding_user_id` / `_verify_onboarding_token`, constant-time compare) replace raw `user_id` in URLs and forms â€” closes IDOR. Empty, malformed, or tampered tokens 403 on both GET and POST. Token secret resolves `APPLYCLING_ONBOARDING_TOKEN_SECRET` â†’ `APPLYCLING_SESSION_SECRET` should become the GA direction; the old UI password fallback is legacy cleanup.
- **Web session auth:** `applycling/auth.py` provides PBKDF2-SHA256 password hashes and HMAC-signed session tokens. `SessionMiddleware` in `applycling/ui/__init__.py` is fail-closed in hosted Postgres mode: requires `APPLYCLING_SESSION_SECRET`, refuses `APPLYCLING_NO_AUTH`, and redirects unauthenticated browser requests to `/login`. `APPLYCLING_ADMIN_USER_ID` gates `/admin`; `/admin/invite` creates active users with generated passwords.
- **MCP server:** `applycling/mcp_server.py` exposes pipeline as MCP tools via FastMCP. `mcp serve` (stdio), `mcp config` (client JSON). Shipped tools: `add_job` (MCP-T1), `list_jobs(limit, status)` and `get_package(job_id)` (MCP-T2), `update_job_status(job_id, status, reason)` via `jobs_service.set_job_status()`, `interview_prep(job_id, stage, model, provider)` writes interview_prep.md, and `refine_package(job_id, feedback, artifacts, cascade, model, provider)` versions before overwriting (MCP-T3). - **MCP alpha installer:** `scripts/setup_mcp.sh` supports the public `curl | bash` path and attaches the interactive `applycling setup` wizard to `/dev/tty`, because prompt input otherwise sees piped stdin and aborts with "Input is not a terminal".
- **Hosted workbench user scoping:** `jobs_service.run_pipeline(job_id, user_id=...)` must build `PipelineContext.from_user_id()` and pass it into `pipeline.run_add_notify()`. Without that explicit context, `run_add_notify()` uses local `data/profile.json` / `data/resume.md`, which caused a 2026-05 alpha incident where a hosted UI generation used the wrong person's profile data.
- **Hosted Telegram scoping:** `_run_scoped_pipeline()` in `applycling/ui/routes.py` must call `run_add_notify(url=ctx.job_url, context=ctx, persist_job=ctx.persist_job)`, not `run_add(ctx)`. Merge refuses running pipelines, moves source jobs/runs/artifacts to the target, fills missing target profile/resume/channel fields from the source, then soft-deletes the source after clearing unique identity fields.

## Key Patterns

- **Thin harness, fat skills:** Python is boring plumbing. Intelligence lives in `.md` files.
- **Library-first design:** `pipeline.run_add()` is the public contract. CLI is a thin wrapper.
- **Deterministic pipeline:** User intent is always "URL â†’ package". No agent routing needed yet (context-based resolvers are planned).
- **Escaped braces in skills:** `{{` and `}}` render as literal `{`/`}` after `str.format`.
- **Conditional logic stays in Python:** Skills have no `if/else`. Caller pre-computes strings, passes as inputs.
- **MCP parity:** When a new pipeline capability ships, the corresponding MCP tool ships in the same PR. Pipeline and MCP surface must not drift.

## Known Gotchas

- **Skill name mismatch:** Loader raises `SkillError` if frontmatter `name` doesn't match directory name.
- **Profile summary header:** Must be `## PROFILE` (all caps) to match format template.
- **`storage.save_config()` merges:** Never call with partial keys unless merging is intent.
- **`_clean_llm_output()` required:** Strips code fences, preamble, leaked prompt markers from all LLM output.
- **`_profile_header_markdown()` builds static header:** Constructs the name/contact block from `profile.json` for resume output. Never let the LLM generate this section â€” always use the pre-computed static header.
- **Anthropic API key in repo .env:** That key is for the applycling pipeline only â€” never use it for Hermes subagent delegation, ad-hoc API calls, or any non-pipeline purpose. Hermes has its own keys in `~/.hermes/`.
